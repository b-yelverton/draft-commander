  <action>
Add mobile glass overrides INSIDE the existing `@media(max-width:599px)` block (line 114-140). Insert these rules at the end of the block, just before the closing `}` on line 140.

**Add these rules inside the existing @media(max-width:599px) block:**

```css
/* Mobile glass reduction */
:root{--glass-blur:8px}
```

Then, also inside the same `@media(max-width:599px)` block, add a nested `@supports` block to remove glass from .section-hdr entirely on mobile:

```css
@supports (backdrop-filter:blur(1px)){
  .section-hdr{-webkit-backdrop-filter:none;backdrop-filter:none;background:var(--surface)}
}
```

**IMPORTANT check — nested @supports inside @media:** CSS allows `@supports` nested inside `@media`. The browser evaluates both conditions. This is standard CSS3 and has the same browser support as `@supports` itself.

**What this achieves:**
1. `--glass-blur:8px` — Reduces the blur radius from 12px to 8px globally on mobile. This affects .hdr and .room-top which use `var(--glass-blur)`. Less blur = less GPU work during scrolling.
2. `.section-hdr` backdrop-filter removed entirely — Multiple sticky section headers in the scrollable board panel create cumulative GPU load. Removing glass from these on mobile preserves smooth scrolling. The solid `background:var(--surface)` fallback keeps them readable.
3. `.modal-overlay` is NOT reduced on mobile — modals are infrequent (not during scroll) and the single 8px blur on a fixed overlay is not a performance concern.
4. `.hdr` keeps glass on mobile (at reduced 8px blur) — only one sticky header, always visible, acceptable GPU cost.

**What NOT to do:**
- Do NOT create a second `@media(max-width:599px)` block. Add rules inside the existing one.
- Do NOT remove glass from .hdr or .room-top on mobile — reducing blur radius is sufficient.
- Do NOT add `will-change` properties — they pre-allocate GPU layers permanently, which is worse for mobile memory.
